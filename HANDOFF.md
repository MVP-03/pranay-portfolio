# Handoff — Pranay Portfolio

Status as of this handoff: **built, content-complete, running locally, not deployed, not yet in git history until this commit.**

## What this is

React + TypeScript + Vite + Tailwind v4 + React Router personal portfolio.
Every non-obvious decision (nav pattern, project card content model, form
states, responsive breakpoints, accessibility rules, pixel-art accent
scope) is recorded in [.plan/portfolio-plan.md](.plan/portfolio-plan.md).
Read that before changing structure or hierarchy — it's the source of
truth for *why* things are built the way they are, not just *what*.

## Run it

```bash
npm install
cp .env.example .env   # add a real Formspree endpoint to make the contact form work
npm run dev             # dev server
npm run build && npm run preview   # production build, served locally
```

Currently served locally only, at whatever port you choose — no live
deploy exists yet. Vercel/Netlify free tier both work with zero config for
a Vite app; that's the natural next step whenever you want it public
(needs a one-time browser login the assistant can't do on your behalf).

## Known tooling gaps hit during this build (informational, not blockers)

- The gstack visual-mockup generator needs an OpenAI API key that isn't
  configured on this machine — mockups were skipped in favor of a
  detailed text-based design review + independent critique pass.
- The gstack `browse` daemon (used for screenshots and PDF rendering) was
  unreliable on this Windows machine — screenshots and the resume PDF
  were generated by calling local Chrome (`chrome.exe --headless
  --screenshot` / `--print-to-pdf`) directly instead.
- Headless Chrome's `--screenshot` flag produces a text-clipping artifact
  at narrow window widths on this machine (verified against a trivial
  control HTML file with no app code — same artifact appeared). Mobile
  responsive behavior should be re-verified with real DevTools device
  emulation, not trusted from those screenshots.

## Design skills now installed

Two design skill packs were added to `.agents/skills/` (universal format,
symlinked for Claude Code, also usable from Codex/Amp/Cline/etc.):

- **`frontend-design`** ([anthropics/skills](https://github.com/anthropics/skills)) —
  guidance for distinctive visual direction: token-system planning (color/
  type/layout/signature element), avoiding templated AI-design defaults,
  copy-as-design-material. Most relevant for the next visual-identity pass
  (e.g., if you want to push the pixel-art accent further, or revisit the
  homepage's "signature element").
- **`emil-design-eng`** ([emilkowalski/skills](https://github.com/emilkowalski/skills)) —
  UI polish and animation decision framework (when to animate, easing,
  duration, spring vs. CSS transitions, `prefers-reduced-motion`). The
  current build has **zero animation/motion** — this skill is the natural
  next pass if you want button press feedback, card hover polish, or
  page-transition treatment, with a clear framework for what's worth
  animating vs. not.

Neither skill has been applied yet — they're installed and ready for
whoever (human or agent) picks up the next design pass.
